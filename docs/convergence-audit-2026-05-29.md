## Usability

Current strengths:
- CLI help is grouped by mode and option category.
- CLI help now explicitly explains that loopback capture device ids come from `devices --source=loopback`.
- CLI help now also makes device-id/backend coupling explicit:
  - capture device ids should match the selected capture backend/source
  - render device ids should match the selected render backend
- CLI help now also makes monitor-off render semantics explicit:
  - `--render-device-id` is ignored when `--monitor=off`
  - `--monitor=off` disables the render pipeline and skips render-device validation
- `devices` mode is available and now prints stable per-device lines with:
  - explicit `CAPTURE_DEVICE` / `RENDER_DEVICE` prefixes
  - quoted device ids
  - Unicode-safe escaped device names (for example `\u9EA6\u514B\u98CE`)
- `devices` mode now also supports an opt-in human-readable path:
  - `--device-name-format=native`
  - default output remains escaped/stable for scripts
  - the real CLI/native output path now writes readable Unicode device names successfully in the current PowerShell/file-driven validation flow
- `devices --source=loopback` now makes an important WASAPI loopback semantic explicit:
  - loopback capture uses render endpoints as capture sources
  - loopback capture devices are labeled separately from ordinary microphone capture devices
  - loopback capture rows now use a dedicated `LOOPBACK_CAPTURE_DEVICE` prefix for easier human and script extraction
  - the render-backed capture guidance no longer repeats redundantly in loopback device output
- Summary text now makes `follow defaults` semantics more explicit:
  - when enabled, manual device picks are described as inactive
- Summary text now also makes WASAPI loopback capture semantics more explicit:
  - loopback capture device selection is described as render-endpoint-backed
- Summary text now also makes the combined follow-default + loopback case more explicit:
  - default-tracked loopback capture is described as following the current default render endpoint
- Diagnostics text now also makes `follow defaults` semantics more explicit:
  - current capture/render selection is described as default-tracked when follow-default-devices is on
- Summary and diagnostics now share helper-backed wording for follow-defaults semantics, reducing drift between the two user-facing text surfaces.
- Summary now also uses helper-backed wording for the monitor-disabled render note instead of a scattered inline string.
- Summary and diagnostics now both use helper-backed wording for monitor-disabled render semantics.
- Detailed quick-probe monitor-disabled render-wave explanation is now also helper-backed instead of staying as an inline string in `app_model.cpp`.
- Diagnostics text now also uses loopback-specific capture labels, so a render-backed loopback endpoint is no longer described under a generic microphone-oriented capture label.
- Loopback summary/backend notes and loopback-specific diagnostics capture labels are now also helper-backed instead of staying as scattered inline strings in `app_model.cpp`.
- CLI quick probe supports overrides for:
  - backend
  - device id
  - source mode
  - rate
  - channels
  - sample type
  - share mode
  - drive mode
  - monitor
  - auto-align
  - dump on/off
  - dump type
  - dump path
  - delay
- CLI matrix supports at least `--matrix-source=mic|loopback|both`.
- CLI matrix now also supports render-backend scoping:
  - `--matrix-render-backend=wasapi|wave|both`
- CLI matrix now also supports capture-backend scoping:
  - `--matrix-capture-backend=wasapi|wave|both`
- CLI matrix now also supports align-mode scoping:
  - `--matrix-align=on|off|both`
- CLI matrix now also supports profile scoping:
  - `--matrix-profile=pcm16-48k-stereo|pcm24-44k-mono|both`
- CLI matrix now also supports WASAPI share-mode scoping:
  - `--matrix-wasapi-share=shared|exclusive|both`
- CLI matrix now also supports delay scoping:
  - `--matrix-delay=0ms|120ms|both`
- CLI matrix now also supports capture/render buffer-pair scoping:
  - `--matrix-buffer=cap40-ren40|cap80-ren120|both`
- Those delay/buffer filters are now locked at three layers:
  - CLI parse/usage coverage in `probe_cli_test`
  - scoped matrix generation coverage in `app_model_text_test`
  - real CLI/script coverage in convergence and hardware-validation entrypoints
- Quick and matrix outputs both expose rich result summaries.
- Quick probe output now also reports the requested capture/render device ids, which makes explicit `device-id` validation runs easier to audit.
- Matrix rows now also report the requested capture/render device ids, which makes explicit loopback/device-targeted matrix runs easier to audit.
- Matrix summaries now also break out device-selection failures explicitly:
  - `CAPTURE_DEVICE_FAIL`
  - `RENDER_DEVICE_FAIL`
- Matrix summaries now also break out startup-stage failures explicitly:
  - `CAPTURE_START_FAIL`
  - `RENDER_START_FAIL`
- Matrix diagnostic hints are now surfaced in the convergence entrypoint itself, not just buried inside full probe text.
- The explicit-device loopback matrix view now surfaces a render-device clustering hint when backend/device mismatch dominates the failures.
- The focused WASAPI-capture + WAVE-render loopback view now surfaces a dedicated hint when capture-missing and unsupported-mode dominate together.

  - users are explicitly directed to `devices --source=loopback`
  - the failure explains that loopback capture expects render-backed endpoints

  - choose a render device from `devices` for the same render backend
  - or omit `--render-device-id`

- `devices` mode is now stable, but escaped non-ASCII names are less friendly than native rendering.
- CLI still lacks matrix-side overrides for many dimensions that quick probe already supports.

- low to medium
